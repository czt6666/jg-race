# Dataset2 数据报告与算法建议

## 1. 任务口径

Dataset2 是时序图未来链接预测 / 候选重排序任务。每条测试样本给定 `src, time, c1...c100`，模型需要输出 100 个候选 `dst` 的交互概率，评测按正样本在 100 个候选中的排名计算 MRR。

因此这个任务的核心不是预测一个绝对概率，而是在每一行的 100 个候选里把真正会发生的 `dst` 排到尽量靠前。提交值必须是 `[0, 1]` 之间的小数，但 MRR 只看相对排序。

## 2. 数据规模与时间范围

训练集：

- 行数：2,261,283
- 列：`src, dst, time, split`
- `split=0`：1,995,088 行，时间为 2000-04-12 到 2009-12-28
- `split=1`：266,195 行，时间为 2009-12-29 到 2011-01-29
- 训练集整体时间：2000-04-12 到 2011-01-29
- 唯一时间戳：3,904 个
- 时间戳基本按天离散，唯一时间戳相邻间隔中位数为 86,400 秒

测试集：

- 行数：153,420
- 列：`src, time, c1...c100`
- 时间范围：2011-01-30 到 2012-01-13
- 唯一时间戳：349 个
- 测试时间紧接训练集之后，没有穿越到训练区间内部

测试集月度分布比较均匀：

- 2011-02 到 2011-12 每月大约 1.1 万到 1.5 万条
- 2012-01 只有 4,142 条，因为只覆盖到 1 月 13 日

## 3. 图结构：严格二部图

Dataset2 是严格二部图：

- 训练 `src` 唯一数：12,708
- 训练 `dst` 唯一数：50,640
- `src` 与 `dst` 交集：0
- 总节点数：63,348
- `src` ID 范围：110,374 到 139,634
- `dst` ID 范围：1 到 110,369
- `dst.max=110369`，`src.min=110374`，中间只差 5 个 ID，基本是两个连续编号空间

这说明不要把它当一般同构社交图处理。更合理的建模视角是用户-物品二部图，或者序列推荐：

- `src` 类似用户、作者、会话、主体节点
- `dst` 类似物品、论文、网页、被交互对象
- 预测任务是给定用户和时间，从候选物品里重排序

## 4. 边重复与新边特征

训练集中唯一有向边数为 2,211,275，占总交互的 97.79%。也就是说：

- 按 `(src,dst)` 看，重复边只占约 2.21%
- 绝大多数交互是新边
- 单条 `(src,dst)` 边出现次数中位数为 1
- 99.9% 的边出现次数不超过 2
- 最大重复次数也只有 2

更关键的是，以 `split=0` 作为历史、`split=1` 作为验证未来：

- `split=1` 正样本中，`dst` 全部在 `split=0` 出现过
- `split=1` 正样本中，`src` 全部在 `split=0` 出现过
- `split=1` 正样本中，重复历史边比例为 0

结论：验证阶段不是在考“这个用户以前点过什么还会不会再点”，而是在考“老用户对已知物品产生新连接”。简单记忆历史边几乎没有直接命中价值。

不过测试候选分布和验证正样本分布存在明显差异，见下一节。

## 5. 测试候选：冷启动候选极多

测试集：

- 测试行数：153,420
- 测试唯一 `src`：2,180
- 测试唯一候选 `dst`：110,368
- 所有测试 `src` 都在训练 `src` 中出现过
- 测试候选中，只有 50,639 个候选 ID 在训练 `dst` 出现过，占唯一候选的 45.88%
- 按候选槽位计算，已知 `dst` 占 46.41%，训练未出现的冷启动 `dst` 占 53.59%

每行 100 个候选里：

- 已知 `dst` 数量中位数：46
- 25% 分位数：43
- 75% 分位数：50
- 90% 分位数：53
- 最少：25
- 最多：69

这非常重要：如果模型只会给训练出现过的 item 打有效分，对未出现 item 给统一低分，那么当真实正样本是冷启动 item 时，MRR 会很差。反过来，如果给所有冷启动 item 随机偏高分，又会压低已知正样本。

所以 Dataset2 的提分关键之一是“已知 item 与冷启动 item 的分数校准”。

## 6. 测试行与 `(src,time)` 多事件

训练集中同一个 `(src,time)` 常常对应多个 `dst`：

- 训练唯一 `(src,time)` 数：787,143
- 平均每个 `(src,time)` 约 2.87 条交互
- 50.89% 的 `(src,time)` 组有多条交互
- 82.90% 的训练行落在多交互 `(src,time)` 组里
- 最大的 `(src,time)` 组有 4,229 条边

测试集也有类似结构：

- 测试唯一 `(src,time)` 数：64,831
- 平均每个 `(src,time)` 约 2.37 条测试行
- 42.62% 的 `(src,time)` 组有多条测试行
- 75.75% 的测试行落在多行 `(src,time)` 组里
- 最大的测试 `(src,time)` 组有 247 行

同一个 `(src,time)` 下，不同测试行的候选集合几乎不重复：

- 测试组内候选去重率均值：99.88%
- 大部分组内候选完全不重叠
- 最大组的候选并集可达 22,124 个

这说明测试数据很可能把同一个用户同一天的多个未来交互拆成多条 100 候选重排序样本。算法上可以共享同一个 `(src,time)` 的用户表示，不必重复计算。

## 7. 度分布与稀疏性

`src` 度分布极不均匀：

- 中位数：14
- 75% 分位：94
- 90% 分位：430
- 95% 分位：967
- 99% 分位：2,587
- 最大：12,574

`dst` 度分布也长尾：

- 中位数：10
- 75% 分位：26
- 90% 分位：80
- 95% 分位：181
- 99% 分位：696
- 最大：3,234

测试 `src` 不是普通用户，而是明显偏活跃的子集：

- 测试 `src` 历史交互数中位数：1,037
- 25% 分位：394
- 75% 分位：2,181
- 90% 分位：3,999
- 最大：12,574

这对算法很有利：测试源节点都有较长历史，因此序列推荐模型可以充分利用用户历史；但由于真实边多数是新边，不能只靠“历史 item 复现”。

## 8. 时间维度特征

数据按天聚合，很多同源交互发生在同一天：

- 对同一个 `src` 的相邻交互时间差，中位数为 0 秒
- 75% 分位：1 天
- 90% 分位：5 天
- 95% 分位：9 天
- 99% 分位：45 天

测试查询距离该 `src` 最后一次训练交互的间隔：

- 最小：1 天
- 10% 分位：40 天
- 25% 分位：94 天
- 中位数：187 天
- 75% 分位：282 天
- 90% 分位：344 天
- 95% 分位：567 天
- 99% 分位：1,316 天

这说明短期兴趣和长期兴趣都重要：

- 对最近活跃用户，近期窗口很可能有效
- 对长时间未活跃用户，需要长期用户画像、全局热门度和时间衰减

## 9. 候选生成分布

测试候选出现次数相当均匀：

- 候选出现次数中位数：138
- 75% 分位：147
- 90% 分位：155
- 95% 分位：160
- 99% 分位：174
- 最大：460

候选 ID 覆盖几乎整个 `dst` 编号空间。按 ID 区间看，测试候选相比训练 `dst` 更接近均匀覆盖，导致大量训练未出现的新 item 被塞进候选。

训练 `dst` 的 ID 分布并不均匀：

- 1-10k：166,599 条训练边
- 10k-20k：77,365 条
- 20k-40k：124,869 条
- 40k-60k：527,036 条
- 60k-80k：600,909 条
- 80k-100k：549,269 条
- 100k-120k：215,236 条

测试候选在 1-110k 上更均匀，因此仅凭训练热门度会系统性偏向已知密集 ID 区间，冷启动 item 会被压低。

## 10. 候选与用户历史的直接关系

每行 100 个测试候选中，已经和当前 `src` 交互过的 item 数：

- 均值：1.49
- 中位数：1
- 75% 分位：2
- 90% 分位：4
- 95% 分位：6
- 99% 分位：10
- 最大：21
- 56.54% 的测试行至少有一个候选是该 `src` 的历史交互 item

但验证集显示未来正样本不是历史重复边。因此对于这些“用户历史 item”：

- 不能简单打最高分
- 它们可以作为用户兴趣的上下文
- 如果候选本身已被用户交互过，反而可能应该轻微降权，除非比赛隐藏正样本包含重复边

## 11. 对现有代码的判断

更贴合 Dataset2 的模型：

- SASRec 系列更自然，因为 Dataset2 是二部图，测试 `src` 历史长，任务像序列推荐。
- DyGFormer 可以用，但如果只做 pairwise BCE，会被 100-way MRR 误导。
- 任何验证都应该使用 99 个负样本模拟 100-way，而不是 1 个负样本。

需要特别注意：

- 输出必须是 100 个 `[0,1]` 分数。
- rank-average 集成如果输出 `0..99` 分数，不符合提交格式；需要归一化或使用 score-average。
- 由于 MRR 只看排序，可以用校准后的单调分数，但仍要压到 `[0,1]`。

## 12. 实用算法建议

### 12.1 首选路线：SASRec + 100-way 验证

推荐把 `13SASRec_wd.py` 作为主线，而不是从最早的 DyGFormer 版本继续堆。

理由：

- 图是严格二部图，本质更像用户-物品推荐。
- 测试 `src` 历史很长，序列模型可以吃到大量行为上下文。
- 真实边几乎都是新边，SASRec 的 item embedding 与序列兴趣匹配比记忆历史边更合适。
- 代码里 `13SASRec_wd.py` 已经使用 `VAL_NEG=99`，验证口径正确。

建议配置：

- 使用全量 `split=0+1` 训练最终模型。
- 验证调参时用 `split=0` 训练、`split=1` 验证。
- 保持 100-way 验证，不看 pairwise MRR。
- 负样本数不要太小，`K=15/31` 比 `K=3/5` 更贴近测试。
- 保留 weight decay，防止已知 item 分数过度极化。

### 12.2 候选内重排序，而不是全库召回

测试已经给了 100 个候选，因此模型只需要对候选重排序。可以把特征和模型都做成 candidate-aware：

对每个候选构造以下特征：

- SASRec 模型分数
- item 全局热门度
- item 最近 30/90/180/365 天热门度
- item 是否训练出现过
- item ID 区间或归一化 ID
- 当前 `src` 历史长度
- 当前 `src` 距离最后一次交互的时间间隔
- 候选是否在当前 `src` 历史中出现过
- 候选与当前 `src` 历史 item 的共现相似度

然后训练一个轻量的 ranker，例如：

- Logistic Regression
- LightGBM / XGBoost LambdaRank
- CatBoostRanker
- 简单 MLP

如果环境不方便装树模型，可以先用线性融合：

`score = a * sasrec + b * recent_pop + c * global_pop + d * cold_score + e * cooccur - f * already_seen`

再用 `split=1` 100-way 模拟验证调权。

### 12.3 冷启动 item 校准

测试候选中 53.6% 是训练未出现 item。对这些 item，普通 embedding 模型无法学习具体表示，常见行为是给相同默认 embedding 或随机 embedding，导致排序很不稳。

可行策略：

1. 区分已知 item 和冷启动 item。
2. 已知 item 用模型分数为主。
3. 冷启动 item 不要统一打最低分；给它们一个受控的中间分数。
4. 冷启动分数应根据当前行已知 item 的分数分布校准。

推荐从简单方案开始：

- 对每行取已知 item 分数的中位数 `median_known`
- 冷启动 item 分数设为 `median_known + small_jitter`
- jitter 可由候选 ID 的 hash 生成，保证稳定且可复现
- 冷启动整体位置用验证或本地假设调参，例如在已知 item 中位数附近、略低或略高

这个策略比“所有冷启动全低”更稳，也比“所有冷启动随机 0.5”更不容易压垮已知正样本。

### 12.4 时间窗口热门度很值得加

验证 `split=1` 正样本对历史窗口覆盖率很高：

- 最近 30 天出现过的 item 覆盖 70.46% 的验证正样本
- 最近 90 天覆盖 85.83%
- 最近 180 天覆盖 93.10%
- 最近 365 天覆盖 96.89%
- 最近 730 天覆盖 98.95%

因此 item 时间热度是强特征。建议给每个候选加：

- 全局历史 count
- 最近 30 天 count
- 最近 90 天 count
- 最近 180 天 count
- 最近 365 天 count
- 时间衰减 count：`sum(exp(-(t_query - t_edge) / tau))`

这些特征对冷启动 item 无能为力，但对已知 item 的相对排序很有帮助。

### 12.5 共现 / ItemCF 作为强基线

因为测试 `src` 历史很长，可以做 item-based collaborative filtering：

对候选 item `c`，计算它与用户历史 item 集合的相似度：

- `score(c) = sum_{h in recent_history(src)} sim(c, h) * decay(time_h)`
- `sim(c,h)` 可用共同 src 数、Jaccard、cosine、BM25 权重
- 历史只取最近 50/100/200 个 item，避免超慢

这个方法对“新边”比重复边记忆更有用，因为它推荐的是与用户历史相似但没交互过的 item。

实现时注意：

- Dataset2 边数 226 万，不算小但还能做稀疏倒排。
- 先用最近窗口和高频 item 限制共现表大小。
- 对超热门 item 做 IDF 降权，避免所有人都推荐同一批热门 item。

### 12.6 同一个 `(src,time)` 共享表示

测试里 75.75% 的行属于重复 `(src,time)` 组。SASRec 或 DyGFormer 推理时应缓存同一 `(src,time)` 的用户表示：

- 对相同 `(src,time)` 只算一次 user embedding
- 然后对该组所有候选批量打分

这不改变 MRR，但可以显著省推理时间，允许跑更多模型或更大模型。

### 12.7 集成方式

推荐集成以下互补分数：

- SASRec 主模型分数
- SASRec 不同随机种子 / 不同 `K` / 不同 weight decay
- 时间热门度分数
- ItemCF 共现分数
- 冷启动校准分数

集成时建议：

- 对每行内部做 rank 或 z-score 归一化
- 最终输出前压到 `[0,1]`
- 如果用 rank average，不能直接输出 0..99，应转换为 `[0,1]`，例如 `(100 - avg_rank) / 99`

## 13. 最小可落地方案

如果只做一版实用提分方案，建议按这个顺序：

1. 用 `13SASRec_wd.py` 作为主模型，确保验证是 100-way MRR。
2. 训练 2-3 个种子或不同 `num_neg` 的 SASRec。
3. 生成测试分数。
4. 对测试候选加冷启动校准：冷启动 item 放到每行已知 item 分数中位数附近。
5. 加一个 item 时间热门度分数，与 SASRec 做行内 rank 融合。
6. 输出前检查 shape 为 `[153420, 100]`，分数范围为 `[0,1]`。

这个方案不追求最新论文，但非常贴合 Dataset2 的数据特征，预计比单纯 DyGFormer BCE 或 pairwise 验证更稳。
