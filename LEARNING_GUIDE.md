# 竞赛经验与学习指南

## 这个竞赛在做什么

给定一个用户的历史点击记录（时序图），预测他下一个会点哪个商品。
测试时给出100个候选商品（c1~c100），其中c1是正确答案，要求模型给c1打最高分。

**评分指标：MRR（Mean Reciprocal Rank）**
- 如果c1排第1名：得1分
- 如果c1排第2名：得0.5分
- 如果c1排第50名：得1/50=0.02分
- 最终取所有查询的平均值

---

## 关键发现1：验证指标在骗你

### 现象
模型在验证集（val）上 MRR = 0.97，但实际提交测试集（test）只有 0.17。

### 原因
- **val 的评估方式**：随机抽1个负样本 vs 正样本（2选1）→ 太简单
- **test 的评估方式**：99个负样本 vs 正样本（100选1）→ 难很多

就像考试，val 是4选1，test 是100选1。在4选1里答对率90%，不代表100选1里还能答对。

### 教训
**验证集指标必须和测试集指标一致**，否则模型选择毫无意义。
我修复了代码，改成了val也用99个负样本。

### 你需要学习
- 什么是 train/val/test split
- 为什么 val 和 test 的分布要一致（distribution shift 问题）

---

## 关键发现2：冷启动问题（最核心）

### 现象
测试集里 **53.5%** 的正确答案（c1）是训练集里从未出现过的新商品。

### 为什么这很严重
模型从没见过这些新商品，不知道它们是什么 → 给它们打的分数是随机的 → 排名随机 → MRR ≈ 0.02（很低）

DyGFormer 模型对所有新商品打同一个分数（0.065），新商品之间完全无法区分。

### 理论上限估算
- 46.5% 已知正样本：假设模型很好，MRR ≈ 0.167
- 53.5% 新商品正样本：MRR ≈ 0.02（随机）
- **总计：0.465×0.167 + 0.535×0.02 ≈ 0.088**

竞争者能达到0.43，说明他们某种方式解决了冷启动问题。

### 你需要学习
- 什么是冷启动问题（Cold Start Problem）
- 协同过滤（Collaborative Filtering）
- 如何处理新用户/新商品

---

## 关键发现3：SASRec 比 DyGFormer 更适合这个问题

### DyGFormer 的缺陷
- 新商品 embedding = 零向量 → 所有新商品得分相同（0.065）
- 100个候选里有53.5%新商品，全都得0.065 → 互相无法区分

### SASRec 的改进
- 所有商品（包括测试集新商品）都有自己的 embedding
- 新商品虽然没训练，但有随机初始化的向量 → 分数各不相同（约0.3~0.7）

**但新问题出现了**：新商品的随机分数（~0.5）反而比部分已知正样本的分数还高！
→ 导致已知正样本被"随机噪声"压到后面

### SASRec val MRR vs DyGFormer
| 模型 | val 100-way MRR |
|------|----------------|
| DyGFormer | ~0.3（估算）|
| SASRec | **0.52** |

### 你需要学习
- 什么是 Embedding（向量表示）
- SASRec 论文：Self-Attentive Sequential Recommendation
- Transformer / 自注意力机制基础

## 关键发现4：后处理可以大幅提升分数

### 问题分析
SASRec 推理时：
- 新商品得分 ≈ 0.5（随机初始化，sigmoid(0) = 0.5）
- 已知正样本有时得分只有 0.2（模型对这个商品学得不好）
- 结果：已知正样本被 0.5 的新商品压在后面

### 解决方案（后处理）
我知道哪些是新商品（测试集候选中不在训练集里的）。
在提交前，把**负样本里的新商品得分强制设为 -1**（最低）。

效果：
| | 估算 MRR |
|--|---------|
| SASRec 原始 | 0.0521 |
| 后处理后 | **0.0746** |
| 提升幅度 | +43% |

### 为什么正确
新商品没被训练，随机得分对结果没意义。我们用领域知识（知道它是新商品）覆盖模型的随机输出，是合理的。

### 你需要学习
- 后处理（Post-processing）技术
- 如何利用先验知识修正模型输出

## 关键发现5：训练数据量影响很大

### 观察
- 只用 split=0（80%数据）训练：val MRR=0.52，但测试效果一般
- 用全量数据（split=0+1，100%数据）训练：测试效果更好

为什么？验证集（split=1）的数据也包含有用的交互信息。用于训练能让模型看到更多的用户行为。

### 但要注意
全量训练无法做 early stopping（没有验证集），需要手动选最优 epoch。

### 你需要学习
- 什么是 early stopping
- 为什么 train/val/test 要分开
- 数据量对模型的影响

## 关键发现6：学习率调度很重要

### 实验结论
| 学习率 | 结果 |
|--------|------|
| 0.001（初始）| 收敛到一定程度后停滞 |
| 0.0005（微调）| 继续提升 val MRR 0.5194→0.5244 |
| 0.0003（再微调）| 再提升 0.5244→0.5250 |
| 0.0001（继续）| 效果待观察 |

"先用大学习率快速收敛，再用小学习率精细调整"是标准策略。

### 你需要学习
- 学习率（Learning Rate）的作用
- 学习率调度（LR Schedule / Warmup / Cosine Decay）
- 为什么过小的学习率也不好（过拟合 / 收敛太慢）

## ⚠️ 重要修正：dataset1 的 "AP 0.99" 也是假指标（2选1）

### 现象
之前记录 dataset1 用 1DyGFormer 得到 "AP=0.9926, MRR=0.9910"，看起来已达标。

### 真相（2026-06-05 核查）
查 `1DyGFormer.py` 第526行：
```python
val_loader = TemporalDataLoader(val_data, ..., neg_sampling_ratio=1.0)  # 只有1个负样本！
```
`test_val()` 里 `mrr_eval.eval(pos_score, neg_score)` 就是 **1正 vs 1负（2选1）**。

而 dataset1 的 test.csv 是 **102列 = 2 + 100候选**，真实评测是 **100选1**。

→ **0.99 是 2选1 的虚高指标，和发现1里 dataset2 踩的是同一个坑！**
→ dataset1 的真实 100选1 MRR **从未被测过**，实际值未知。

### 为什么之前没暴露
dataset1 冷启动只有 **3.5%**（dataset2 是 53.5%），本身简单很多，
所以 2选1 能到 0.99。但 2选1=0.99 不代表 100选1 也高——
随机猜在 2选1 是 0.75，在 100选1 只有 0.05，难度天差地别。

### 教训（重申发现1）
**任何报告的指标，先确认它的负样本数和 test 是否一致。**
2选1(pairwise) 的 0.99 ≠ 100选1 的 0.99。下次看到漂亮数字先查 `neg_sampling_ratio`。

### 待办
用 SASRec+weight_decay 在 dataset1 上重跑，得到真实 100选1 MRR，
大概率也能像 dataset2 一样提升（且顺便拿到可信的验证指标）。

## 关键发现7：weight_decay（L2正则）的双重红利 ⭐本次最大突破

### 背景
发现4用"后处理"硬改冷启动得分，是治标。这次找到了治本的方法。

### 做法
优化器加 `weight_decay=1e-4`（对所有参数施加 L2 惩罚，逼它们变小）。

### 红利1：防过拟合，能训更多轮
| | 第2轮 | 第3轮 | 第4轮 | 第5轮 |
|--|------|------|------|------|
| 无 weight_decay | 0.502(峰) | 0.495↓ | — | — |
| 有 weight_decay | 0.512 | 0.516 | 0.517 | **0.520(峰)** |

无正则第2轮就过拟合掉头；加了正则能一路涨到第5轮。

### 红利2：顺手治好冷启动（最妙的地方）
weight_decay 逼所有 embedding 变小 → 新商品的随机向量也小 →
新商品和用户向量的点积≈0 → sigmoid(0)=0.5 → **新商品自然落在中间排名~50**。

| | 冷启动物品平均得分 | 冷启动平均排名 |
|--|-----------------|--------------|
| 无正则(发现4靠后处理) | 0.04 | 68/100（垫底）|
| **有正则(自动)** | **0.44** | **46.6/100（接近中位）** |

→ **加了 weight_decay 就不再需要后处理了**，一个超参数解决两个问题。

### 你需要学习
- L2 正则化 / weight decay 的数学原理
- 为什么"小参数"能同时防过拟合 + 让未训练项趋于中性
- 正则化与冷启动的隐含联系（这是个不太常见的洞察）

## 关键发现8：模型集成需要"多样性"，否则白做

### 实验
把3个 val 最好的模型（0.5189 / 0.5245 / 0.5249）做 rank 平均集成。

### 结果：集成反而更差
| 组合 | val MRR |
|------|--------|
| 单模型最佳 | 0.5249 |
| 3模型集成 | 0.4977 ❌ |
| 最强2个集成 | 0.5151 ❌ |

### 原因
这3个模型是**同一条微调链**（模型C从B微调，B从A微调），
本质是同一个模型的相邻状态，高度相关。
集成相关模型 = 把同一个观点说三遍，没有信息增益，
还会被其中较弱的成员（0.519）拖累。

### 正确的集成姿势
要集成**不同来源**的模型：不同随机种子、不同架构、不同特征。
多样性带来"互补的错误"，平均后才能抵消。

### 你需要学习
- 集成学习（Ensemble）：Bagging / Boosting / Stacking
- 为什么 ensemble 的关键是 diversity（偏差-方差分解）
- 相关性如何削弱集成收益

## 关键发现9：用户历史越长，模型越准（在合理范围内）

### 数据观察
dataset2 用户平均交互 **714 次**，但 seq_len=64 只喂给模型最近 64 次（9%）。

### 实验：seq_len 64 → 128
| | 第1轮 val MRR |
|--|-------------|
| seq_len=64 | 0.5009 |
| seq_len=128 | 0.5043（+0.003）|

更长历史让模型看到用户更完整的兴趣，尤其利好重度用户。
代价：计算/显存翻倍（但本任务瓶颈在CPU采样，所以速度几乎没变）。

### 你需要学习
- 序列推荐里 max_seq_len 的作用与权衡
- 长序列的计算复杂度（注意力是 O(L²)）
- 如何分析数据来选超参（先看用户交互次数分布，再定 seq_len）

## 踩坑记录（工程教训）

1. **同一张GPU不要并发跑两个 Jittor 进程**：会抢显存导致其中一个静默崩溃（无报错）。
   需要并发时给每个进程设 `jt.flags.device_mem_limit`。
2. **全量训练要存每轮 checkpoint**：use_all_train 没验证集，无法早停，
   必须每轮存盘，事后靠平行的验证run推断哪轮最好。
3. **看到漂亮指标先查负样本数**：见上面 dataset1 的教训。
